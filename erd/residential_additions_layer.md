# ResidentialAdditions - new Cumulative_Accounting REST layer

Designed 2026-05-15. Normalization of the analyst's `FINAL RES SUMMARY 2012 to 2025.xlsx`. Path-to-REST for the year-by-source matrix that `residential-additions-by-source.html` currently embeds as 3 inline JS arrays.

## Source

- **File**: `data/from_analyst/FINAL RES SUMMARY 2012 to 2025.xlsx`
- **Sheet**: `Summary`
- **Shape**: row-by-metric grid (14 metric rows x 14 year columns). Converter extracts the 7 atomic source rows (5 Added + 2 Removed) and pivots to long form.
- **Cadence**: snapshot, regenerated by the analyst on the cumulative-accounting cycle. Replace the xlsx and re-run the converter when a refresh arrives.
- **As-of date**: 2026-05-15

## Identity preserved

For every year:
```
Net Change Residential Units = SUM(Added.Units) + SUM(Removed.Units)
```
The Removed.Units are stored negative so a simple sum gives the net. Converter validates this identity against the xlsx's Net Change row and reports a warning on mismatch. 2026-05-15 snapshot: passes all 14 years (2012-2025).

## Normalized schema (proposed SDE table)

```sql
CREATE TABLE dbo.ResidentialAdditions (
    ObjectID     INT IDENTITY(1,1) NOT NULL PRIMARY KEY,
    Year         INT          NOT NULL,
    Direction    VARCHAR(10)  NOT NULL,   -- 'Added' | 'Removed'
    Source       VARCHAR(20)  NOT NULL,   -- 'Allocations' | 'Bonus Units' | 'Transfers' | 'Conversions' | 'Banked' | 'Converted'
    Units        INT          NOT NULL,   -- signed: positive for Added, negative for Removed
    SourceRow    VARCHAR(80)  NOT NULL,   -- provenance: original xlsx row label
    AsOfDate     DATE         NOT NULL,
    LoadedDate   DATE         NOT NULL,
    SourceFile   VARCHAR(120) NOT NULL,
    CONSTRAINT UQ_ResidentialAdditions UNIQUE (Year, Direction, Source, AsOfDate)
);
```

### Why long instead of wide

7 source/direction combinations x 14 years x N future cycles = many rows but uniform shape. Long form lets new sources or directions appear without altering schema (e.g., a future "From Density Bonus" addition source becomes a new row, not a new column). The dashboard pivots client-side for the facet charts.

### Why publish atomics only (not Net Change / Total Built)

Net Change is `SUM(Added) + SUM(Removed)`; Total Built is the running cumulative of Net Change starting from the 2012 baseline (47,550). Both are derivable client-side. Publishing them as separate rows risks the layer disagreeing with itself if the underlying atomics shift. Keep the layer normalized; let consumers aggregate.

If a downstream needs the 2012 baseline + cumulative trajectory, surface it as a constant in the consumer (already done in `tahoe-development-tracker.html` for PDH-derived totals) or publish a tiny companion `ResidentialBuilt_Baseline` table (one row: `Year=2012, Units=47550`).

   - `SOURCES` + `REMOVED` collapse into a single fetch from layer 11; reshape client-side into per-source year arrays for the facets

## Intermediate state (this commit)

The converter and JSON are in place. The dashboard will be repointed to read from `data/processed_data/residential_additions.json` (a REST-STAGED snapshot) immediately so the inline arrays are eliminated. When layer 11 is live, the URL constant in the dashboard flips from the local JSON to the REST endpoint - no other dashboard code changes.

## What this corrects

Today the dashboard's facet charts come from manually-typed JS arrays. Risks:
- Stale: arrays only refresh when a developer edits the file
- Drift: array values can be silently different from the xlsx if someone edits one without the other
- Hidden: no automated check that the dashboard agrees with the analyst's tally

After layer 11 publishes, the dashboard always agrees with the SDE staging, and the SDE staging always matches the analyst's xlsx (via the converter's identity check). Three steps in the chain, all reproducible.
