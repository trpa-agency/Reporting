## Root Causes Fixed (This Session)

### Root Cause 1 — El Dorado Split-Format Duplicate Rows
**Fixed in a prior session; confirmed working**

El Dorado County switched APN suffix format from 2-digit to 3-digit in 2018. A coworker manually handled some parcels by entering the same parcel twice in the CSV — once with the 2-digit APN, once with 3-digit. The ETL's El Dorado normalization then produced two rows for the same `(APN, Year)`. The csv_lookup dict (last-write-wins) silently overwrote the real value with zero in some cases.

**Fix:** Groupby-max dedup after El Dorado fix in S02. This run removed 490 duplicate rows.

**Note for coworker:** The residential CSV should have one row per APN. El Dorado format normalization is handled entirely by the ETL — having both 2-digit and 3-digit forms in the CSV creates this conflict.

---

### Root Cause 2 — Genealogy Missing `change_year` Values
**Fixed today**

The ETL's genealogy step (S02b) silently skips rows without `change_year`. As of the April 14 run, 2,397 rows were skipped. Without `change_year`, units under retired APNs are never redirected to their current parcel.

**Fix:** Created `scripts/detect_change_years.py` — queries OUTPUT_FC and SOURCE_FC to find when each old APN last appears and when the new APN first appears. Recommends `change_year = last_year_old + 1` with confidence ratings.

From the output, **91 HIGH/MEDIUM_OFFBYONE pairs** were promoted directly into `apn_genealogy_master.csv` as `is_primary=1`. An additional **16 ALREADY_HANDLED pairs** (coworker had already split units correctly in the CSV) were added as `is_primary=0` for record completeness.

`apn_genealogy_master.csv` grew from 1,848 → 1,958 rows. READY substitutions: 804 → 806.

### Root Cause 4 — Post-Genealogy Duplicate Rows in df_csv
**Fixed today — resolved the entire 2018–2025 excess deficit**

When S02b substitutes `old_APN → new_APN`, it only skips if the target APN already has a **non-zero** unit count. If the target has a zero-unit placeholder row for that year, the conflict check passes and the substitution creates a **second row** for `(new_APN, Year)` in df_csv — the original 0-unit row and the newly substituted non-zero row.

The csv_lookup dict comprehension (last-write-wins) then wrote whichever row iterated last — in some cases the 0-unit row won, silently zeroing out the real unit count in the FC. S6 QA's `df_csv.groupby.sum()` counted both rows, so CSV_Total appeared correct while FC_Total was wrong — making the bug invisible in per-APN inspection and only visible as an unexplained aggregate deficit.

**Affected APNs (before fix):**

| APN | County | Extra deficit/yr | Years affected |
|-----|--------|-----------------|----------------|
| `123-032-18` | WA | 4 | 2018–2025 |
| `029-620-016` | EL | 1 | 2018–2025 |
| `015-304-033` | EL | 2 | 2018–2020 |
| `098-360-001` | — | 1 | 2018–2020 |
| `080-162-018` | — | 1 | 2019–2020 |

**Fix:** Added a second groupby-max dedup step in `s02_load_csv.py` immediately after `s02b_genealogy.run(df_csv)`, before building csv_lookup. This run removed 90 duplicate rows and eliminated the entire 2018–2025 excess deficit.
