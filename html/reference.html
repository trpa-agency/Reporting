Data & Architecture Reference

Where the data comes from, what each REST layer holds, and the row-level schema roadmap. For "which dashboard answers my question," go to index.html instead.
This is the technical reference for the TRPA cumulative-accounting dashboards. Inventory of data sources, REST layer catalog, refresh pipelines, and the canonical-schema roadmap. Built for data engineers / analysts / auditors who need to know where the numbers come from.

Active set: 7 dashboard entry points (★ marks the primary view in each conceptual track). Every active visualization reads live REST against the Cumulative_Accounting service (17 layers, 0-16) or its TRPA-hosted siblings; no analyst xlsx in the runtime loop.

Layout variants: 5 of 7 dashboards have alternative layouts (Option A = canonical / Option B = newer treatment / Option C exists for tracker only). Each variant carries an eyebrow "switch to Option X layout" link. Decided-against variants move to html/_archive/ and surface as compact cards on index.html + are listed in the Archived section below. Repo: trpa-agency/Reporting

Where the data comes from #

Three sources feed the dashboards. The first two are durable; the third is the bottleneck the roadmap below is designed to eliminate.

1. TRPA Enterprise Geodatabase → REST (live, row-level)

Parcel Development History (layer 0), Tahoe APN Genealogy (layer 1), Residential Unit Inventory (layer 2), Allocations 2012 Regional Plan (layer 4), and AllParcels are all native Enterprise GDB feature classes / tables published through maps.trpa.org. Refreshed by the upstream ETL pipelines: PDH by parcel_development_history_etl/ (this repo), genealogy by the four-source merge documented in GENEALOGY.md.

2. LT Info webservices → staged nightly to REST (live, mostly row-level)

Corral-origin data flows out exclusively through LT Info's JSON web services. stage_ltinfo_allocations.py is a generic 4-pipeline runner that pulls each service into a Cumulative_Accounting staging layer nightly via Task Scheduler:

  • GetDevelopmentRightPoolBalanceReport → layer 5 (pool balances; ~50 pools)
  • GetDevelopmentRightTransactions → layer 6 (transaction log, ~3,000 rows; 2014-04 to 2026-05; pre-Accela not digitized)
  • GetBankedDevelopmentRights → layer 7 (current bank state per APN; 71% drift vs Corral pci, see banked_reconciliation_summary.md)
  • GetTransactedAndBankedDevelopmentRights → layer 8 (APN-transaction junction, ~5,200 rows)

The older Development_Rights_Transacted_and_Banked REST service is being deprecated; same data now in layers 6/7/8.

3. Analyst spreadsheets → normalized → SDE → REST (snapshot, not row-level)

Layers 9-16 are snapshots of the analyst's hand-assembled spreadsheets. Each layer is a tidy long-form pivot of a specific xlsx, loaded into SDE, published as a REST table, and consumed directly by the dashboards. Per-layer publishing memo + converter linked below:

These 8 snapshot layers are the path-of-least-resistance solution. They unblock the dashboards today, but each one inherits the analyst's spreadsheet workflow as an external dependency: a refresh requires the analyst regenerating the xlsx, the converter being re-run, the SDE table being re-loaded, the REST service being republished. That's the bottleneck the row-level roadmap below is designed to eliminate.

Forward path: row-level schema #

Every snapshot layer (9-16) could become a SQL view derived from row-level facts. Once the underlying tables exist for all four commodities, the analyst's xlsx becomes a validation check, not a data delivery.

The architectural proposal (design memo + DDL + view definitions):

  • canonical_row_level_schema.md - 8-entity row-level schema (AllocationRight, AllocationEvent, etc.), gap analysis per commodity, 5-phase migration path (forward-build with dual-write, then backfill RES → RBU → CFA/TAU in order of difficulty), organizational/funding callouts
  • canonical_schema_ddl.sql - SQL Server DDL for the 8 entities + 2 lookups with FKs, indexes, CHECK constraints (e.g. CK_Right_StatusLocation enforces status agrees with where the row sits), seed data
  • canonical_views.sql - CREATE OR ALTER VIEW statements deriving every current snapshot layer (10/11/12/13/14/15/16 + 9) from the 8 tables

Why this matters

  • Lineage becomes traceable: "where did this unit come from?" becomes a SELECT walking AllocationEvent history.
  • Project rollups become answerable: "which allocations were used for Sugar Pine?" returns rows.
  • Construction tracking becomes uniform: ConstructionStatus on every AllocationRight means Tracker section iii becomes a query, not a snapshot.
  • Banked stops drifting: BankedDevelopmentRight becomes a view over the transaction log, not a stored field. The 71% drift problem dissolves structurally.
  • New commodities are additive: a future Stormwater Credit is a new row in the Commodity lookup + events that flow, not a new xlsx schema.
  • Analyst role shifts: from "produce the snapshot" to "validate the database matches expectations." The xlsx becomes a check, not a delivery.

The honest gap

Three pre-Accela datasets aren't recoverable from any digitized source:

  • Per-allocation CFA / TAU / RBU history (1987 - mid-2010s)
  • Per-event RES transactions (1986 - 2013, pre-Accela)
  • Per-event CFA / TAU events (pre-Accela)

These have to be synthesized from totals plus plausible-distribution assumptions, tagged IsSynthesized=1 in the schema so future analysts know which rows are interpretation vs record. Going forward (post-cutover), every new event lands in the row-level schema natively. Earlier diagnosis of this gap in path_to_normalized_rest.md.
