## 3. Pre-ETL: backfill_jurisdiction.py

Before the full ETL run, `backfill_jurisdiction.py` was run to populate `JURISDICTION` and `COUNTY` fields in the source FC via spatial join to `Boundaries/FeatureServer/10`.

- One centroid per unique APN built from the source FC
- Pass 1 (WITHIN) + Pass 2 (CLOSEST ≤ 100m) join to the Jurisdictions service
- County names converted to 2-char codes: Washoe → WA, El Dorado → EL, Placer → PL, Douglas → DG, Carson City → CC
- Jurisdiction names converted: City of South Lake Tahoe → CSLT; unincorporated county areas receive the county code
- These fields now carry into `Residential_Parcels_History` via the s01 FC copy on every ETL run

### ⚠ Check 3 — Duplicate APN × Year: 115 ROWS FLAGGED

**115 duplicate APN × Year rows** were detected in the output FC. This is a new issue — prior runs showed 0 duplicates.

These duplicates likely originate from the source FC refresh. If the parcel history dataset was updated and introduced duplicate records for some parcels in some years, those duplicates carry through the s01 copy into the output FC. A duplicate APN × Year means the ETL populated two rows with the same unit count for the same parcel and year, which will cause double-counting in any aggregation or analysis.

**This must be resolved before the output FC is used for analysis.** Steps:
1. Open `QA_Duplicate_APN_Year` in ArcGIS Pro to inspect which APNs are duplicated and in which years.
2. Trace back to the source FC (`Parcel_History_Attributed`) to determine if the duplicates exist there or were introduced by the copy.
3. If duplicates are in the source: coordinate with the parcel history dataset maintainer to remove them.
4. If duplicates appeared in the copy process: investigate s01 for any unintended row multiplication.

### Check 5 — FC Units Not in Raw CSV (NEW)

This check identifies FC rows with `Residential_Units > 0` whose APN does not appear in the raw (un-transformed) CSV for that year. Results are categorized by the transformation that explains why the APN differs.

| Category | Rows | APNs | Units | Description |
|---|---|---|---|---|
| EL_DORADO_FORMAT | 154,943 | 19,416 | 188,199 | Expected — FC 3-digit El Dorado APN, raw CSV still uses 2-digit |
| CROSSWALK_REMAP | 1,223 | 308 | 1,937 | Expected — units assigned via spatial centroid join in s03 |
| GENEALOGY_REMAP | 359 | 57 | 1,166 | Expected — units moved to successor APN via s02b substitution |
| **UNKNOWN** | **549** | **182** | **551** | **⚠ Investigate — no known transformation explains this** |

The `EL_DORADO_FORMAT` category (154,943 rows) is expected and correct — the El Dorado APN fix is working as intended; the FC uses the 3-digit format post-2018 while the raw CSV still uses 2-digit.

The **UNKNOWN category (549 rows / 182 APNs / 551 units)** increased from 127 APNs (prior run) to 182 APNs (+55). These are FC parcels with units where the ETL's known transformation paths (El Dorado format, genealogy, crosswalk) do not explain why the raw CSV doesn't list that APN for that year. Possible causes:
- APNs that received units via the crosswalk but were not logged in `QA_APN_Crosswalk` (edge case)
- APNs that appear in both the old and new CSV but under slightly different formatting
- Genuine data inconsistencies in the source FC introduced by the refresh

Open `QA_FC_Units_Not_In_CSV` in ArcGIS Pro, filter to `Category = 'UNKNOWN'`, and spot-check against the raw CSV and parcel history FC.

## 7. Recommended Next Steps

### Priority 1 (Urgent) — Resolve 115 Duplicate APN × Year Rows

Open `QA_Duplicate_APN_Year` and trace duplicates back to the source FC. The output FC should not be used for analysis until these are resolved. If duplicates exist in `Parcel_History_Attributed`, the dataset maintainer needs to remove them before the next ETL run.

### Priority 2 — Investigate 40 PARCEL_NEW APNs (950 units)

30 of the 40 PARCEL_NEW APNs are newly added to the CSV this run and have no geometry in the FC or AllParcels service. Check each against county assessor records to confirm:
- Are these valid APNs that should appear in the parcel history?
- Do they exist under a different APN format (e.g., formatting differences not caught by the El Dorado fix)?
- Are any of them data entry errors in the CSV?

### Priority 3 — Investigate UNKNOWN Category in Check 5 (182 APNs, +55 vs. prior run)

The 55 new UNKNOWN APNs in `QA_FC_Units_Not_In_CSV` may be related to the source FC refresh. Filter to `Category = 'UNKNOWN'` and cross-reference against the 191-row change in the FC and the 255 new CSV APNs to determine the source.

### Priority 4 — Investigate UNKNOWN Lost APNs (167 APNs, +56 vs. prior run)

The UNKNOWN category in `QA_Lost_APNs` grew by 56 APNs. Some likely belong to the 255 new CSV APNs that have partial FC history. Filter `QA_Lost_APNs` to `Issue_Category = 'UNKNOWN'` and sort by `Total_Units_CSV` descending to prioritize.

### Priority 5 — Fill 163 Missing change_year Entries in Genealogy Master

163 primary genealogy rows were skipped for the third consecutive run. Most are Washoe County pre-2012 renumbering events. Bulk-entering `change_year = 2012` for the `1318-xx-xxx-xxx` style APNs would activate them on the next run.
